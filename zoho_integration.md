The OAuth client secret and refresh token must never be committed to this repository. They are stored only in `/etc/protracklite.env` on production, whose permissions are restricted to `0600`.

Abhishek's one-time authorization code was exchanged successfully. Zoho issued a refresh token, token refresh was verified, and the Zoho People Leave API accepted read requests. The production environment contains:

```env
ZOHO_CLIENT_ID=...
ZOHO_CLIENT_SECRET=...
ZOHO_REFRESH_TOKEN=...
ZOHO_ACCOUNTS_URL=https://accounts.zoho.in
ZOHO_PEOPLE_URL=https://people.zoho.in
```

## Verified access and leave types

The production verification matched **22 of 24** active ProTrack employee email addresses to Zoho People. Two employee accounts must have their email addresses reconciled before their leave can synchronize.

Zoho currently returns these four employee-eligible leave types:

| Zoho leave type | Zoho ID | Unit | Proposed ProTrack mapping |
|---|---:|---|---|
| Earned Leave | `24413000000183927` | Days | Planned |
| Leave Without Pay | `24413000000159083` | Days | Unpaid |
| Maternity Leave | `24413000000159075` | Days | Not currently represented in ProTrack |
| Work From Home | `24413000002552035` | Days | Not currently represented in ProTrack |

Zoho does not currently return eligible leave types corresponding to ProTrack **Sick**, **Casual**, or **Comp Off**. HR must create/assign those leave types in Zoho People or approve an alternate mapping before automatic leave creation is enabled. ProTrack must not silently map these categories to Earned Leave.

## Previous API permission issue

The original token generated by `shantanu.singh@solulever.com` refreshed successfully, but Zoho People returned:

```text
7077 — Sorry! your role is not allowed to access APIs.
```

Enable API access for the integration account in Zoho People:

```text
Settings
→ Manage Accounts
→ User Access Control
→ Function Based Permissions
→ API Access
```

The production integration now uses Abhishek's Super Administrator client instead, and this blocker is resolved.
